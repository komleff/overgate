# План прокачки агентного пайплайна (v3.3)

**Дата:** 2026-04-13
**Статус:** Утверждён оператором
**Авторы:** Claude Opus 4.6 (v1) → GPT-5.4 (ревью 1) → Claude Opus 4.6 (v2) → GPT-5.4 (ревью 2) → Claude Opus 4.6 (v3) → GPT-5.4 Instant + Thinking (ревью 3) → Claude Opus 4.6 (v3.1) → GPT-5.4 Thinking (ревью 4) → Claude Opus 4.6 (v3.2) → GPT-5.4 Thinking (ревью 5) → Claude Opus 4.6 (v3.3)
**Контекст:** Анализ 4 проектов и 6 конкурирующих фреймворков. Пять раундов кросс-модельного ревью.

---

## Цель пайплайна

Снять с не-программиста-оператора ручную координацию контроля качества, сохранив Zero Trust и cross-model review. Оператор не читает код, опирается на бинарные вердикты, а качество подтверждается тестами, 4-аспектным ревью и разделением ролей.

**Основная проблема:** не недостаток отдельных агентов, а недостаток жёсткой автоматизации quality loop.

---

## Инварианты пайплайна

Механизмы, которые нельзя убирать без нового явного обоснования. Pipeline Audit проверяет их соблюдение.

1. **Все review-pass публикуются в PR.** Push, изменивший код после последнего review/verify-артефакта, без нового отчёта = незавершённый цикл. Промежуточные push во время разработки (до первого review) не подпадают.
2. **Один владелец публикации (PM).** Субагенты возвращают findings, PM публикует.
3. **Нет merge без fresh review на текущем commit.** Review привязан к commit hash.
4. **Любое незавершённое замечание имеет адрес.** fix now / defer to Beads (с ID) / reject with rationale.
5. **Проверки централизованы через `/verify`.** Один gate, одна точка изменения.
6. **PM не искажает findings ревьюверов.** Агрегация и структурирование — да. Изменение смысла — нет.
7. **Merge — отдельное решение оператора.** Агенты доводят до merge-ready, не мержат.

**Обоснование вынесения инвариантов в отдельную секцию:** Рекомендация ревью 2. Без явного списка pipeline-audit сложнее, drift быстрее появляется, а правила размазаны по разным документам.

## Приоритизация

- **P0** — устраняет реальные сбои или закрывает системные failure modes
- **P1** — усиливает качество до кода или раскрывает готовые возможности
- **P2** — стратегические улучшения, требующие обкатки

## P0: Устранение системных сбоев

### 0.1 Merge-readiness lock + `/finalize-pr`

**Проблема:** PM оптимизирует под завершение задачи. Любое необязательное правило будет проигнорировано. Чеклист с формулировкой «PM обязан» — это рекомендация, не enforcement.

**Решение:** Новый скилл `/finalize-pr <PR_NUMBER>` — единственный разрешённый способ объявить PR готовым к merge.

**Логика `/finalize-pr`:**

1. Зафиксировать текущий HEAD commit hash.
2. Проверить: `/verify` прошёл на этом commit? Если нет — **СТОП**.
3. Проверить: последний внутренний review-pass привязан к этому commit? Если нет — **СТОП**.
4. Проверить: если tier = Sprint Final — существует ли внешний review-pass на этом commit? Если нет — **СТОП** («External review обязателен для Sprint Final»).
5. Проверить: если был `CHANGES_REQUESTED` от внешнего ревью — есть ли повторный review-pass после фиксов на этом commit? Если нет — **СТОП**.
6. Проверить: все замечания имеют статус (fix now / defer с Beads ID / reject с обоснованием)? Если есть замечания без статуса — **СТОП**.
7. Warning: если >50% замечаний = defer — вывести предупреждение оператору.
8. Если все проверки пройдены — опубликовать финальный комментарий:

**Фаза 1 (до внедрения 1.4):**
   ```
   ## ✅ Готов к merge
   Commit: <hash>
   Verify: ✅
   Internal review: ✅ (commit <hash>)
   External review: ✅ (commit <hash>) / N/A
   — PM (Claude Opus 4.6), /finalize-pr
   ```

**Фаза 2 (после внедрения 1.4):**
   ```
   ## ✅ Готов к merge
   Commit: <hash>
   Verify: ✅
   Internal review: ✅ (commit <hash>)
   External review: ✅ (commit <hash>) / N/A
   Unresolved findings: 0
   Deferred to Beads: <list of IDs>
   — PM (Claude Opus 4.6), /finalize-pr
   ```

**Enforcement:**
- Прямой `gh pr comment` с текстом «готов к merge» / «ready to merge» перехватывается hook-ом и блокируется с сообщением: «Используй /finalize-pr».
- В таблице «Защита от самообмана»: «Всё прошло, можно мержить» → «Нет. Только /finalize-pr. Ручной "готово" заблокирован».

**Правило обязательности external review:** Для Sprint Final — external review обязателен, `N/A` не допускается. Для Standard и ниже — external review опционален, `N/A` валиден.

**Emergency override:** `/finalize-pr --force` — только по явной команде оператора или при зафиксированной ошибке самого скилла. PM не может инициировать `--force` самостоятельно. Требует: указание причины, пометка в PR `⚠️ Force finalize: [причина]`. `--force` не отменяет обязательность отчёта — PM вручную публикует состояние каждого gate в PR-комментарии. Bypass только сломанного механизма автоматической валидации, не самих проверок.

**Двухфазное внедрение:** На фазе 1 (шаг 5a) `/finalize-pr` работает без triage validation (шаги 1–5 логики). На фазе 2 (шаг 5b, после внедрения 1.4) добавляются шаги 6–7 (проверка статусов замечаний и defer-abuse warning).

**Где разместить:** `.claude/skills/finalize-pr/SKILL.md`

**Обоснование принятия:** PIPELINE_ADR: «для ИИ-агентов soft constraints = no constraints». Чеклист в v2 был правильным по содержанию, но неправильным по форме — оставался рекомендацией. `/finalize-pr` превращает его в hard gate.

**Обоснование commit binding:** Ревью 2 указало на конкретный сценарий поломки: review прошёл → Developer внёс правки → PM не запустил новый review → объявил merge-ready. Привязка к commit hash — единственный способ это предотвратить. Каждый review-pass фиксирует commit hash в PR comment, `/finalize-pr` сверяет HEAD с последним review.

### 0.2 Единый владелец публикации review-pass + защита от искажения

**Проблема:** Допускаются два режима публикации (PM или Reviewer). Серая зона ownership. Дополнительно: PM может «сгладить» или потерять детали findings при консолидации — это нарушает Zero Trust.

**Решение:**

> Любой review-runner возвращает structured findings PM. В PR всегда публикует один владелец прохода — PM/оркестратор.

> PM не имеет права изменять findings ревьюверов. Допустимо: структурирование по 4 аспектам, агрегация дубликатов, добавление triage-статуса. Не допустимо: перефразирование, смягчение формулировок, удаление findings.

**Enforcement для external-review:** Raw output каждого внешнего ревьюера сохраняется в PR comment полностью (или как collapsible `<details>` блок), отдельно от консолидированного отчёта PM. Оператор может сверить.

**Где менять:**
- AGENT_ROLES.md — добавить правило неизменности findings. Исключение для прямого запуска Reviewer оператором (без PM) — Reviewer публикует самостоятельно, т.к. оператор может запускать внешние ИИ-модели напрямую
- reviewer.md — явно: «возвращай результат, не публикуй»
- external-review/SKILL.md — raw output сохраняется в collapsible блоке

**Обоснование принятия:** v2 зафиксировал единого владельца. Ревью 2 указало на вторую проблему: PM может искажать findings. Collapsible raw output — минимальный enforcement, не усложняющий workflow.

### 0.3 Verify как единый обязательный gate

Везде заменить `npm run build && npm run test` на `/verify`. Скилл расширяем при добавлении линтеров, typecheck и т.д.

**Где менять:**
- sprint-pr-cycle/SKILL.md шаг 1.1
- PM_ROLE.md секция 2.2 шаг [3]

**Обоснование принятия:** Verify уже существует. Это унификация с нулевой стоимостью: единая точка изменения, расширяемость, меньше способов пропустить gate.

**Обоснование повышения с P1 до P0 (v2):** Рекомендация ревью 1. Низкозатратная критичная унификация.

### 0.4 Pipeline Audit как регулярный скилл

**Логика `/pipeline-audit`:**

1. Собрать список всех нормативных файлов (.agents/*.md, .claude/agents/*.md, .claude/skills/*/SKILL.md, .claude/rules/*.md, .claude/settings.json).
2. Проверить: одинаковое ли число аспектов ревью упоминается везде.
3. Проверить: совпадают ли имена ролей, веток, команд.
4. Проверить: нет ли формулировок, противоречащих deny-rules в settings.json.
5. Проверить: все ли ссылки между документами валидны.
6. Проверить: инварианты 1–7 отражены во всех нормативных файлах без противоречий.
7. Отчёт: [OK / DRIFT DETECTED] с конкретными расхождениями.

**Триггер:** после каждых 3–5 спринтов; обязательно перед изменением самого пайплайна.

**Где разместить:** `.claude/skills/pipeline-audit/SKILL.md`

**Обоснование принятия:** ADR: drift между документами — один из самых частых failure modes. Pipeline Audit — антивирус против расползания.

### 0.5 Degradation path для external-review (4 режима)

| Режим | Условие | Что происходит | Adversarial diversity |
|-------|---------|----------------|----------------------|
| **A** | API key | GPT-5.4 + GPT-5.3-Codex | Максимальная (cross-model) |
| **B** | ChatGPT login | Один проход дефолтной модели | Снижена |
| **C** | Codex CLI недоступен | Два прохода Claude: стандартный + adversarial | **Degraded mode с имитацией adversarial diversity. Не является cross-model review.** |
| **D** | Автоматические пути недоступны | VS Code GitHub Copilot Agent с ChatGPT-5.4, ручной | Ручная, degraded |

**Режим C:** В отчёте обязательна формулировка: «⚠️ Degraded mode: adversarial diversity имитирована промптом, не обеспечена разными моделями. Не эквивалентно cross-model review».

**Режим D:** В отчёте: «⚠️ Manual emergency mode. Adversarial diversity и воспроизводимость снижены».

**Обоснование уточнения режима C (v3):** Ревью 2: прежняя формулировка «не маркировать как кросс-модельный» — недостаточно явная для честного audit trail.

## P1: Усиление качества и контроля

### 1.1 Контекстная изоляция PM (процедурная)

Правила:
1. После каждой завершённой задачи PM обязан обновить `status.md`.
2. После каждого review-pass PM формирует compact handoff packet.
3. После 3 задач **или** 5 review-итераций по одному PR в одной сессии — PM обязан рекомендовать оператору session reset.
4. В таблице «Защита от самообмана»: «Контекст чистый» → «>3 задач или >5 review-итераций — перезапусти».

**Обоснование порога review-итераций (v3):** Ревью 2: при 5–10 итерациях review основная деградация от длинного review-loop, не от числа задач.

---

### 1.2 Tester gate для Critical

Для Critical tier: PM запускает Tester-субагента → Tester возвращает findings PM (инвариант 2) → PM делегирует Developer'у → повторный `/verify`.

---

### 1.3 Test Design / Verification Contract до кода

Planner обязан создать секцию с acceptance criteria, expected behaviors, edge cases, error cases, инвариантами и списком тестов до merge.

**Enforcement (через reviewer):** Reviewer в аспекте «Качество» обязан проверить:
- Наличие Verification Contract.
- Соответствие тестов контракту.
- Если контракт отсутствует — аспект «Качество» автоматически `CHANGES_REQUESTED`.

**Обоснование enforcement (v3):** Ревью 2: «без enforcement пункт станет декларативным». Проверка через reviewer — минимальный overhead.

---

### 1.4 Протокол triage замечаний

| Статус | Значение | Действие | Валидация |
|--------|----------|----------|-----------|
| **fix now** | Блокирует PR | Developer исправляет, повторный review-pass | — |
| **defer to Beads** | Техдолг | PM создаёт issue | **Валиден только с Beads ID. Без ID — замечание неразрешённое** |
| **reject with rationale** | Не принято | PM фиксирует обоснование | Обоснование обязательно |

**Защита от defer-abuse:** Если >50% замечаний имеют статус defer → `/finalize-pr` выводит warning оператору. Не блокировка, но сигнал.

Многократные итерации (5–10 циклов) — штатный режим. Зафиксировать в PM_ROLE.md и PIPELINE.md.

**Обоснование ужесточения defer (v3):** Ревью 2: «defer без Beads ID = потеря замечания». Исторически подтверждено.

---

### 1.5 Автообновление Memory Bank по событию

Предупреждение (не блокировка) при финализации PR, если status.md не обновлён. Это сознательно мягкий первый шаг — если практика покажет, что предупреждения игнорируются, повысить до блокировки.

## Критерий успеха

- PM не может объявить merge-ready в обход `/finalize-pr`
- `/finalize-pr` проверяет commit binding: review привязан к текущему HEAD
- Все review-pass публикуются одним владельцем, raw findings сохраняются
- Sprint Final не блокируется при недоступности Codex CLI (4 режима)
- Pipeline Audit проверяет инварианты, находит 0 расхождений
- Verification Contract существует до кода, reviewer проверяет покрытие
- Каждое замечание имеет статус; defer валиден только с Beads ID
- Regressions и reopened отслеживаются в метриках
- Многократные итерации review — штатный режим
