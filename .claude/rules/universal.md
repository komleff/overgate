# Универсальные правила

## Язык

- Комментарии в коде — только на русском
- Идентификаторы — на английском

## Git workflow

- Пуш в main запрещён — только ветки и PR
- Merge — только оператор
- Checkpoint-коммиты перед опасными изменениями

## Качество

- После изменений: запусти сборку и тесты проекта (через `/verify`). Пример (U2): `npm run build && npm run test`; `dotnet test`.
- Новый код без тестов не мёржится — PR с кодом без тестов будет отклонён Reviewer (doc-only PR — исключение)
- Атомарность: одна задача = один фокус
- Работай по задачам — не делай ничего, что не запрошено
- Не "улучшай" стиль, архитектуру, форматирование без запроса

## AI-артефакты

- Не удалять завершённые спеки, планы и другие AI-артефакты без явного запроса оператора
- Устаревшие или завершённые AI-артефакты переносить в `docs/archive/`

## Гигиена корневых инструкций

- Не добавлять временные заметки в `AGENTS.md` / `CLAUDE.md` (для этого есть `.memory-bank/`)
- Предлагать правки оператору перед изменением `AGENTS.md` / `CLAUDE.md`
- После редактирования — проверить файл на дублирование и противоречия

## Канон-решения

- Перед задачами по архитектуре, физике, сети, ГД, нарративу или монетизации читать `docs/architecture/ADR-INDEX.md`.
- Если предлагаемое решение противоречит записи реестра — не «улучшать по ходу». Стоп, новый ADR со статусом `supersedes`, потом код.
- Каждый новый ADR в `docs/architecture/` обязан содержать блок «Отклонённые варианты» с причинами отклонения. ADR без этого блока в реестр не вносится.
- Шаблон нового ADR — `docs/architecture/ADR-TEMPLATE.md`.
- Устаревшие ADR не удаляются. Им меняется статус на `superseded` со ссылкой на заместителя.

## Эскалация

- При расхождении требований — сообщить оператору
- При конфликте источников правды — сообщить оператору
