# Правила тестирования

> **Baseline-числа ниже — пример из U2 reference; замени на свои.** Методология (TDD, AAA, что тестировать, запрещённые паттерны) generic — её менять не нужно. Конкретные счётчики тестов, warning-baseline и стек-специфика — `<PLACEHOLDER>`'ы, которые заполняются под фактический стек проекта.

## Подход

- TDD: RED → GREEN → REFACTOR
- AAA-паттерн: Arrange / Act / Assert
- Один тест = один сценарий

## Обязательно тестировать

- Доменная логика и ядро (формулы, расчёты, инварианты состояния) — основным тест-фреймворком проекта
- Клиентская prediction/smoothing/UI-логика (если применимо) — клиентским тест-фреймворком
- Интеграционные тесты: реальные зависимости вместо моков; за отдельным флагом (например `<INTEGRATION_FLAG>`, по умолчанию выключены)
- Edge-cases: 0, null, граничные значения, экстремальные сценарии (для сетевых систем — экстремальные RTT: 50ms / 200ms / 500ms)
- Доменные инварианты, которые легко регрессируют (фиксируй их тестом со ссылкой на источник-доктрину проекта)

## Baseline counts (для отслеживания регрессий)

> Заполни под свой стек. Назначение раздела — детектить рост числа тестов/warnings и появление новых типов warnings. Шаблон ниже сохраняет смысл U2-реализации; конкретные числа — пример.

- **Build warnings baseline: `<WARNING_BASELINE>`** (вся солюшн/сборка, как в `/verify`; зафиксируй эталонную cold-команду измерения и состав по кодам warnings). **Правило (аудит baseline, не каждый прогон):** `> <WARNING_BASELINE>` warnings при cold-сборке = регрессия (новые чинить или явно подавлять с обоснованием). Обычный warm-`/verify` НЕ аборти́тся на warnings (это tech-долг) — сравнение с baseline делается отдельным cold-прогоном (периодически / в CI / при подозрении на рост).

- Основной фреймворк: `<EXPECTED_SERVER_TESTS>` (например `~843 passed / 6 skipped`; зафиксируй конфигурацию прогона и дату).
- Клиентский фреймворк: `<EXPECTED_CLIENT_TESTS>` (например `~319 passed / 7 skipped`; integration-блоки skipped без `<INTEGRATION_FLAG>`).

## Запрещено

- Тесты, которые всегда проходят (бессмысленные `assert(true)`)
- Отключение или удаление существующих тестов без явного обоснования и согласования
- Тестировать реализацию вместо поведения
- Мокать реальные зависимости в integration tests (используй настоящие — это смысл интеграционных)
- Игнорировать failed тесты (PreToolUse hook на `git commit` блокирует commit при падении тестов, если в проекте есть `package.json`)
