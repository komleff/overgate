# ADR-NNNN — <короткая тема>

> Шаблон для нового архитектурного или продуктового решения. Заполняется по образцу существующих ADR проекта. После заполнения запись добавляется в `docs/architecture/ADR-INDEX.md`.

## Контекст

Что происходило до решения. Какая проблема, какие силы давили, какие ограничения уже были зафиксированы (доктрина, ранее принятые ADR, требования продакшена). Один-два абзаца. Без литературных украшений.

## Решение

Что именно решено. Формулируется одним абзацем. Должна быть однозначно проверяемая формулировка: можно посмотреть на код или документ и сказать «соблюдается» / «не соблюдается».

## Отклонённые варианты

Обязательный блок. Без него ADR не считается принятым — он перестаёт быть фильтром от «пережитков».

|---|---|
| Альтернатива 1 | Причина отклонения с конкретикой: метрики, инциденты, доктринальные ограничения. |
| Альтернатива 2 | Причина отклонения. |
| Альтернатива 3 | Причина отклонения. |

## Триггер пересмотра

Условия, при которых ADR должен быть переоткрыт. Без явного триггера ADR живёт бессрочно. Примеры:

- Конкретная метрика вышла за порог.
- Появилась новая платформа или зависимость, конфликтующая с решением.
- Изменилась сторонняя доктрина, на которую опирается ADR.
- Приходит явное решение оператора отменить ADR.
