# ADR-0000 — Реестр канон-решений как фильтр от пережитков

## Контекст

Проект (на примере U2) живёт месяцами с одним оператором и несколькими ИИ-агентами (Claude, Codex, Gemini). Контекст между сессиями передаётся через `.memory-bank/` (Cline-style Memory Bank). Memory Bank хорошо проявил себя на короткой и средней дистанции — оперативный фокус, прогресс, текущая ветка. Но архитектурные и продуктовые канон-решения в нём не задерживаются: `activeContext.md` обновляется при каждой смене фокуса, а старые принципы туда не переписываются — они «и так известны».

В результате новый агент, открывший задачу, читает `activeContext.md` + `progress.md`, идёт в спеку и предлагает решение, которое уже было обсуждено и отклонено в доктринах, brand- или decision-документах проекта (в U2 — Doctrine, Brand Bible, Patches). Оператор тратит силы на повторное «убийство» уже похороненных идей. Это классические **пережитки** — концепции, потерявшие свой контекст, но продолжающие всплывать в обсуждениях.

Конкретные примеры пережитков, выявленные на практике (здесь — из U2, иллюстрация механики; в твоём проекте будут свои):

- «Семья дома у игрока» как нарративный якорь — была в раннем варианте Patches №3, удалена как несовместимая с антитревожным дизайном. После удаления концепция «родного региона игрока» осталась по инерции и тоже была удалена в Patches №4.
- «Лети за свежими ценами» как торговая разведка — конфликтует с Slow Light, но регулярно всплывает у новых агентов.
- «Полная физика трастеров как в Star Citizen» — отклонена в спеке выбора полётной модели, но без явного фильтра агенты предлагают её снова.

Источник идеи реестра — статья «Гигиена проекта при работе с ИИ» (habr.com/ru/sandbox/283242/, см. `.agents/REFERENCES.md`). Статья описывает четыре слоя проектной памяти: живое состояние, снимок, стабильные знания, архив решений. Memory Bank закрывает первые два слоя. ADR-реестр закрывает четвёртый — архив решений с причинами отклонения альтернатив.

## Решение

В проекте вводится **реестр канон-решений** в `docs/architecture/` со следующей структурой и правилами.

**Структура.**

- `docs/architecture/ADR-INDEX.md` — однострочные записи по принятым канон-решениям. Источник истины при сверке «было ли это уже решено».
- `docs/architecture/ADR-NNNN-<slug>.md` — отдельный файл на каждое решение со сквозной нумерацией. Полная аргументация, отклонённые альтернативы, последствия, триггер пересмотра.
- `docs/architecture/ADR-TEMPLATE.md` — шаблон нового ADR. Все новые ADR создаются через копию шаблона.

**Обязательные поля каждого ADR.**

- Контекст: что происходило до решения, какие силы давили, какие ограничения уже были.
- Решение: однозначно проверяемая формулировка.
- **Отклонённые варианты**: обязательный блок. Без него ADR не считается принятым — реестр перестаёт быть фильтром от пережитков.
- Последствия: что меняется в коде, процессе, документации.
- Триггер пересмотра: условия, при которых ADR должен быть переоткрыт.

**Маршрут чтения.**

- `AGENTS.md`, секция «Первое, что делать перед задачей»: ADR-INDEX обязателен к чтению для задач, затрагивающих архитектуру, физику, сеть, ГД, нарратив или монетизацию.
- `.claude/rules/universal.md`, секция «Канон-решения»: дублирующее правило в rules-файле, который Claude Code загружает автоматически.
- Memory Bank остаётся коротким и оперативным; долгоживущие решения переезжают в реестр.

**Жизненный цикл ADR.**

- ADR не удаляются. Устаревшие переходят в статус `superseded` со ссылкой на ADR-заместитель в поле `superseded_by`.
- Заместитель указывает в поле `supersedes` номер вытесненного ADR.
- Полностью отменённый ADR без замены получает статус `deprecated` с причиной отмены в новой версии файла.

**Что нельзя делать.**

- Если предлагаемое решение противоречит записи реестра — стоп. Сначала ADR-пересмотр (новый ADR), потом код. Не «улучшать по ходу».
- Не выносить в реестр оперативные детали и параметры баланса — для них есть spec'и и ship config'и. Реестр фиксирует только то, что нельзя оспаривать без пересмотра.
- Не дублировать содержимое Memory Bank в ADR. ADR — про долгоживущие принципы, Memory Bank — про текущее состояние.

## Отклонённые варианты

| Вариант | Почему отклонён |
|---|---|
| Оставить всё в Memory Bank, не заводить ADR | Memory Bank — оперативный слой, `activeContext.md` ограничен по объёму и обновляется часто. Долгоживущие решения там тонут. |
| Хранить канон-решения только в доктринах/decision-документах проекта (в U2 — Doctrine и Patches) | Они работают как описание принципов, но не имеют обязательного блока «отклонённые альтернативы» и не устроены как сквозной реестр. Новый агент не знает, что искать и где. |
| Завести ADR в Beads (issue tracker) | Beads хорош для трекинга задач, но не для долгоживущих документов с цитированием в коде. ADR должны быть в git-истории как файлы, не в БД issue-tracker'а. |
| Завести ADR без обязательного блока «отклонённые альтернативы» | Без причин отклонения новый агент не понимает, почему вариант X плохой, и предлагает его снова. Главная функция реестра теряется. |
| Линковать ADR из Memory Bank вместо отдельного маршрута чтения в AGENTS.md | Memory Bank — оперативный слой, в нём список ADR быстро устаревает. Маршрут чтения в AGENTS.md и rules стабилен. |
| Делать ADR только для архитектуры, без ГД и нарратива | Проект может быть чувствителен к нарративным/продуктовым пережиткам не меньше, чем к архитектурным (пример U2: «семья дома», «родной регион»). Реестр охватывает все области, где есть канон-решения. |
| Делать ADR-0000 необязательным мета-документом | Без явной фиксации процесса как ADR сам реестр висит без причины существования. ADR-0000 — самореферентная фиксация, чтобы у процесса был тот же фильтр, что и у остальных решений. |

## Последствия

- При проектировании любого решения, попадающего под охват реестра, агент сначала проверяет ADR-INDEX. Если решение конфликтует — пишет новый ADR, не код.
- Каждое канон-решение, ранее зафиксированное в доктринах, патчах, brand-документах, принципах монетизации или decision-документах проекта, переезжает в реестр как отдельный ADR (пример U2: ADR-0001..ADR-0016 в первом проходе).
- Новые ADR создаются через копию `ADR-TEMPLATE.md`. Без блока «Отклонённые варианты» ADR не вносится в INDEX.
- Реестр пересматривается при появлении новой большой темы или при первом обнаружении пережитка, для которого не нашлось ADR — это сигнал, что надо завести новый.
- Memory Bank остаётся коротким. Долгоживущие принципы туда не дублируются — ссылка на ADR в маршруте чтения достаточна.

## Глоссарий

| Термин | Значение |
|---|---|
| ADR | Architecture Decision Record. Документ, фиксирующий принятое решение, его причины и отклонённые альтернативы. |
| Канон-решение | Решение, которое нельзя оспаривать без явного ADR-пересмотра. |
| Пережиток | Концепция, потерявшая свой контекст, но продолжающая всплывать в обсуждениях. |
| Memory Bank | Cline-style набор файлов в `.memory-bank/` для оперативного контекста проекта. |
| Реестр канон-решений | `docs/architecture/ADR-INDEX.md` со списком всех ADR. |
| `superseded` / `superseded_by` | Поля ADR, фиксирующие замену одного решения другим. |
| `deprecated` | Статус ADR, отменённого без замены. |

## Триггер пересмотра

- Процесс не работает: агенты не читают ADR-INDEX даже после правки маршрута чтения. Тогда нужен hook на старт сессии, печатающий индекс.
- Реестр разрастается до неуправляемого размера (>50 активных ADR в одной плоской таблице). Тогда нужна категоризация по областям с подындексами.
- Появляется лучшая практика управления долгоживущими решениями (например, новый формат от индустрии). Тогда — миграция через новый ADR с `supersedes: ADR-0000`.
- Оператор отменяет ADR.
